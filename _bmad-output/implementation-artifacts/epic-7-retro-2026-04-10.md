## Partie 1 — Revue Epic 7

### 1.1 Synthèse de livraison

- **Stories livrées :** 6 (parcours nominal, contexte et blocages, catégories / pesée / qualification, historique et traçabilité, comportement défensif, validation exploitabilité terrain).
- **Ligne directrice tenue :** `Peintre_nano` comme runtime ; vérité métier flux matière, permissions et refus côté `Recyclique` ; checklist PR « Peintre sans métier » réinvestie dans les Dev Notes.
- **Artefact structurant :** `references/artefacts/2026-04-09_01_reception-v2-exploitabilite-terrain-epic7-squelette.md` — registre consolidé (tableau à quatre colonnes, chronologie MCP / stack servie, gaps nommés).

### 1.2 Ce qui a bien fonctionné (systèmes et processus)

- **Réplication du modèle Epic 6 → 7** : story **7.6** comme consolidation (pas une simple « fin de sprint »), avec critères AC alignés sur **6.10** (preuve stack `http://localhost:4444`, trois piliers lint/build/test + registre honnête).
- **Continuité avec la rétro Epic 6** : les actions §2.3 de la rétro 6 (registre tôt, preuves réseau, mapping brownfield, barème create-story) ont été **largement suivies** — mapping réception dans **7.1**, registre vivant dès le squelette, puis consolidation **7.6**.
- **Frontières explicites** : Epic 6 (caisse) vs Epic 7 (réception) vs Epic 8 (sync Paheko) vs Epic 9 (déclaration) documentées dans le registre §6 — réduit le risque de dérive de périmètre pendant l'implémentation.
- **Alignement contrats** : évolution OpenAPI réception (ex. draft **0.1.9-draft** pour historique / export), catalogues CREOS et tests contrat (`creos-reception-nominal-manifests-7-1`, gouvernance OpenAPI).
- **Transposition disciplinée des patterns caisse** : **7.2** calquée sur **6.2** (gate contexte, pytest + Vitest) sans dupliquer la logique métier dans le front.

### 1.3 Difficultés et frottements (sans blâme)

- **Preuves navigateur incomplètes sur sous-ensembles** : le registre note explicitement l'absence de passe MCP sur certaines mutations lignes (`POST/PUT/DELETE`, PATCH poids), export CSV, profil **403** sans `reception.access` — couverture forte en **tests automatisés**, mais **HITL partiel** sur ces points.
- **Pytest « réception » global non vert** : exécution large de `test_reception*.py` → échecs **pré-existants** hors périmètre 7.1–7.5 ; stratégie honnête = **sous-ensemble ciblé** documenté (`test-summary-story-7-6-e2e.md`) — friction récurrente du brownfield (même famille que caisse).
- **Dette schéma nommée** : `poste_reception` **sans** `site_id` — cohérence site / USER documentée en **7.2** ; reste une **dette structurelle** à traiter quand le périmètre multi-site durcit.
- **Clavier-first** : non audité méthodiquement (NFR transverse) — aligné avec le constat Epic 6, pas réglé par Epic 7.
- **Nom de fichier registre** : suffixe historique `…-squelette` conservé malgré document **consolidé** — risque de confusion pour un lecteur neuf (pure lisibilité process).

### 1.4 Apprentissages

1. **Le registre à quatre colonnes** (Validé / Non validé / Hors périmètre / Dette) force l'honnêteté sur « vert tests » vs « preuve stack servie » — complémentaire aux e2e mockés.
2. **La convention `localhost:4444` vs `127.0.0.1`** évite les faux négatifs de preuve ; mériterait d'être rappelée systématiquement dans les briefs Story Runner.
3. **Réutiliser 6.x comme gabarit** (structure story, fichiers tests, reception-client) accélère la cohérence, à condition de **ne pas** confondre domaines (caisse ≠ réception) dans les critères produit.
4. **Isoler pytest ciblé** quand la suite globale est rouge est un **choix documenté**, pas une omission — à condition que la liste des fichiers / classes inclus reste versionnée et reproductible.

|--------|--------|-------------|

### 1.6 Changements de méthode utiles à pérenniser

- **Story finale d'exploitabilité** pour tout flow critique « Convergence 3 » : même exigence que **6.10** / **7.6**.
- **Registre dès 7.1** (ou équivalent sur futurs epics) : évite la dérive entre « code mergé » et « preuve terrain ».
- **Pour chaque gate** : exiger citation des `operation_id` OpenAPI dans la trace (registre §7).

## Partie 2 — Préparation Epic 8 (sync Paheko)

### 2.1 Dépendances satisfaites par Epic 7

- **Flux matière local** exploitable dans la chaîne v2 avec historique ticket, garde-fous contexte, états défensifs — base pour **pousser** ou **corrélater** vers Paheko sans rouvrir la viabilité du wizard réception.
- **Convention de preuve** (registre, localhost:4444, sous-ensemble pytest) réutilisable pour les stories **8.x** lorsque des intégrations réseau réelles seront en jeu.

### 2.2 Risques spécifiques Epic 8

- **Complexité sync / quarantaine / idempotence** : risque de **reporter** de la logique métier dans le front si les états ne sont pas **contratés** OpenAPI + CREOS comme pour la réception.
- **Héritage de la dette pytest** : les gates **8.x** devront décider si on **élargit** progressivement le vert ou si on maintient des **sous-suites** documentées (comme 7.6) avec plan de remise à niveau.
- **`poste_reception` / site** : peut **interférer** avec les correspondances site / caisse / emplacements Paheko (**8.3**) si non traité.

### 2.3 Actions et accords proposés (SMART, sans estimation temporelle)

1. **Renommer ou alias documentaire** le registre Epic 7 — *critère :* un lecteur voit « consolidé » sans lire le corps du fichier (ou lien depuis `references/artefacts/index.md` explicite). **Owner suggéré :** tech writer / pilotage.

2. **Repasser `guide-pilotage-v2.md` §3** pour Epic 6 **et** Epic 7 (jalons, liens registres `_07_` caisse + `_01_` réception). **Owner suggéré :** pilotage.

3. **Avant première story 8.x ready-for-dev** : relire `2026-04-02_05_contrat-minimal-sync-reconciliation-paheko.md` + matrice intégration Epic 1 — *critère :* brief `create-story` **8.1** cite les invariants sync et les `operation_id` cibles. **Owner suggéré :** architecte + SM.

4. **Planifier une passe HITL optionnelle** pour fermer les lignes « Non validé » du registre réception (mutations lignes navigateur, export CSV, 403 profil) si le produit l'exige avant beta — *critère :* registre mis à jour ou décision produit « accepté tel quel » tracée. **Owner suggéré :** PO + QA.

5. **Spike ou ticket** sur `poste_reception.site_id` (ou équivalent de cohérence) — *critère :* décision documentée (accepter / migrer / reporter) avant **8.3**. **Owner suggéré :** architecte + dev backend.

### 2.4 Découverte significative pour la planification

L'**exploitabilité** d'un second flow critique (réception) **renforce** l'hypothèse Epic 6 : la **validation finale par registre** n'est pas du luxe — c'est le **contre-poids** à une suite de tests backend **incomplète au global**. Epic 8 devra **anticiper** la même exigence sur au moins une story de type « validation sync / quarantaine » avec états **visibles** dans Peintre.
