## Partie 1 — Revue Epic 6

### 1.1 Synthèse de livraison

- **Stories livrées :** 10 (parcours nominal, contexte et garde-fous, ticket en attente, remboursement, encaissements spéciaux, actions sociales, clôture locale, corrections sensibles auditées, défensif / sync différée, validation exploitabilité).
- **Ligne directrice tenue :** `Peintre_nano` comme runtime de rendu et de flows ; règles métier, auth, persistance et refus serveur côté `Recyclique`.
- **Artefacts utiles :** manifests CREOS cashflow, widgets et wizards dédiés, suites Vitest `cashflow-*`, registre d'exploitabilité multi-passes avec preuves réseau + UI, captures dans `implementation-artifacts/screenshots/`.

### 1.2 Ce qui a bien fonctionné (systèmes et processus)

- **Rebaselining brownfield-first** après correct course : les slices ne priment plus sur le récit opérateur ; le dashboard `/caisse` réintègre ouverture, variantes et workspace de vente de façon vérifiable.
- **Preuve « servie » structurée** : enchaînement de passes UI documentées (dont MCP `user-chrome-devtools`, **modèle rapide** pour les gates finales), croisement avec lint / test / build, sans confondre vert CI et verdict terrain.
- **Honnêteté sur le périmètre** : clôture de **6.10** avec **gaps nommés** (pas de parité brownfield exhaustive revendiquée, clavier-first non audité méthodiquement, gestionnaire liste sessions non rejoué comme en legacy) — évite la dette d'attente implicite.
- **Couverture fonctionnelle large** : held, remboursement, spéciaux, sociaux, clôture avec step-up, correction bornée, états défensifs et notices « local vs sync » cohérents avec la frontière Epic 8.
- **Corrections transverses révélatrices** : alignement auth (`bearerOrCookie` sur `sales`), robustesse `cash-sessions`, migration de réparation données locale, ajout `transverse.admin.view` dans l'enveloppe pour aligner CREOS et permissions réelles — symptômes systémiques (contrat ↔ runtime ↔ base) plutôt qu'erreurs ponctuelles isolées.

### 1.3 Difficultés et frottements (sans blâme)

- **Découverte tardive de ruptures d'intégration** : premières passes UI ont mis en lumière **500** sur `current` / détail session et **401** sur soumissions vente — écart entre modes d'auth attendus, sérialisation et données réelles vs scénarios de test.
- **Mesure honnête de scénarios edge** : le cas UUID progressif / `sales/held` a nécessité un **protocole explicite** (neutralisation contrôlée de `current` sur une page de test) pour observer le réseau sans confondre artefact de test et produit.
- **Charge cognitive « slice → continuum »** : tant que la navigation et le dashboard ne reflètent pas le workflow brownfield, le risque de fermer trop tôt sur la base des seuls tests e2e isolés reste élevé.
- **Dérive documentaire** : `guide-pilotage-v2.md` §3 conserve une formulation de gate Epic 6 qui ne reflète pas l'état **YAML** ni la clôture **6.10** sous réserves documentées — à réaligner pour éviter deux « vérités » pour les agents.

### 1.4 Apprentissages

1. **Une story de validation terrain (6.10) doit être traitée comme un système de preuves**, pas comme un commentaire final : passes numérotées, tableau checklist, séparation local / Epic 8.
2. **Les échecs sur données critiques (`current`, vente, admin)** signalent souvent une désalignement **OpenAPI / impl / envelope / profil utilisateur** ; la réponse efficace est transversale (front + back + données), pas seulement un patch UI.
3. **« Done » honnête** = critères AC franchis **et** limites explicitement listées ; cela prépare Epic 8 sans geler une fiction de parité legacy.
4. **Les recommandations de la rétro Epic 5** (preuve sur `4444`, alignement statuts) ont été **réinvesties** ; le chaînon manquant était la **granularité** des preuves réseau et la **résilience** aux mêmes classes de bugs sur le domaine caisse.

### 1.5 Dette, risques et sujets à suivre

| Sujet | Nature | Commentaire |
|--------|--------|-------------|
| Parité brownfield complète | Risque produit | Mapping legacy vs v2 reste la référence des écarts structurels. |
| Clavier-first | Dette / non mesuré | Exigence epics ; non couverte par la grille de clôture 6.10. |
| Gestionnaire sessions (liste) | Gap fonctionnel documenté | Détail admin OK ; liste brownfield non équivalente revendiquée. |
| `is_virtual` / KPI virtuels | Limite contrat | Simulation UI OK ; évolution API si le terrain exige la distinction comptable. |
| Convergence 3 (PRD / archi) | Jalon global | Caisse avance ; `data_contract` / `DATA_STALE` à garder comme critère de durcissement, pas comme optionnel décoratif. |

### 1.6 Changements de méthode utiles à pérenniser

- **Gate terrain en passes courtes** : rouge → correctifs → repasse, avec une ligne dans le registre par itération.
- **Exiger pour les flows critiques** : capture ou trace réseau sur la **stack officielle** (docker + port documenté), en complément des tests Vitest.
- **Quand un correct course touche l'epic** : mettre à jour **en même temps** guide de pilotage, story files et YAML pour une seule baseline lisible.
