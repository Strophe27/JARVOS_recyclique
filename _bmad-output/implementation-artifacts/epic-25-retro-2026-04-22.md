### 1.2 Ce qui a bien fonctionné

- **Découpage en deux phases** dans un seul epic numéroté : décisions et artefacts **avant** la montée en charge code — limite les refontes contradictoires sur Paheko / PIN / async.
- **Traçabilité « papier + YAML »** : addendum gel, commentaires `sprint-status`, note readiness — réduit les discussions « on avait dit ».
- **DAG machine** (`epic-25-phase2-dag-2026-04-21.yaml`) + règles dans `epics.md` — permet parallélisme maîtrisé (**25.12** vs **25.7**, etc.) sans casser les prérequis métier.
- **Gates répétables** : pytest par story, **compileall** où pertinent, chaînes **QA doc** pour les stories sans code — prévisibilité pour le Story Runner.
- **Distinction maintenue** entre **levée process** (**25.6**) et **NOT READY** programme PWA — explicitée dans plusieurs livrables ; évite de promettre une PWA sur la base d’un dégel BMAD.

### 1.3 Difficultés et frottements (système, pas personnes)

- **Charge cognitive « epic unique »** : 15 stories variées (doc, infra contractuelle, Python, Peintre, spikes) — le risque est la fatigue de revue ; le DAG et les commentaires YAML compensent mais doivent être **lus** avant promotion de clé.
- **Boucles CR** documentées sur **25.2**, **25.6**, **25.7**, **25.12** : signal que **doc + chemins de tests + `_REL_PATHS`** doivent être complets **avant** premier CR quand l’epic est audit-sensitive.
- **Double statut historique** : à un instant, **`epic-25: done`** pour la tranche **25.1–25.5** puis **réouvert** **in-progress** pour la phase 2 — exige une discipline de lecture des **commentaires** YAML pour ne pas sur-interpréter un export partiel.
- **Alignement slug / titre** : rappels dans YAML sur **25.5** (wildcard dans titre), **25.10** / **25.13** (renommages `versus`) — fragilité des slugs générés vs titres français longs.

### 1.4 Apprentissages

1. **Trois natures de « gate »** : gel process, readiness PWA/programme, gate API P0 — les nommer dans chaque story qui touche au périmètre transverse.
2. **Checklist avant scale** : **25.7** comme pont entre spec normative et preuves — réutiliser le pattern pour tout socle qui risque l’« implémentation par inadvertance ».
3. **Spikes avec bornes** : **25.11** et **25.15** montrent qu’un spike utile **interdit** souvent une livraison complète — l’AC doit lister les **stop criteria** aussi clairement que les livrables.
4. **Rétro Epic 24 absente** : pour les epics « operationnels » (24) suivis d’un socle transverse (25), une **mini-synthèse de continuité** (même 10 lignes) évite la perte de contexte entre rails.

### 1.5 Dette, risques et sujets à suivre

| Sujet | Nature | Commentaire |
|--------|--------|-------------|
| **NOT READY PWA massif** | Programme / readiness | Inchangé : les stories **25.x** ne le lèvent pas ; toute extension kiosque large reste sous **implementation-readiness** et **note 2026-04-20**. |
| **Acceptation humaine ADR** | Gouvernance | Les ADR **25-2** / **25-3** restent des décisions **proposées** jusqu’à arbitrage explicite pour implémentations qui iraient au-delà du train documentaire. |
| **IndexedDB / cache local** | Produit futur | **25.15** explore sans livrer — tout stockage client durable reste à trancher hors spike. |
| **Couverture e2e terrain** | Preuve utilisateur | Beaucoup de valeur est démontrée par **pytest** et contrats ; les parcours réels caisse/kiosque restent une couche **HITL** à planifier selon epic suivant. |
