## Partie 1 — Revue Epic 8

### 1.1 Synthèse de livraison

- **Stories livrées :** 7 — premier slice **bout-en-bout** Recyclique → tentative HTTP Paheko (**8.1**) ; idempotence, retries, statuts explicites (**8.2**) ; correspondances site / caisse / emplacements (**8.3**) ; quarantaine, résolution manuelle, audit des transitions (**8.4**) ; corrélation inter-systèmes et timeline support (**8.5**) ; blocage sélectif des actions finales critiques / **409** politique A1 (**8.6**) ; passe de validation et matrice de preuves (**8.7**).
- **Ligne directrice tenue :** **Recyclique backend autoritaire** ; outbox dans la même transaction que la clôture ; pas d’appel Paheko depuis Peintre ; OpenAPI versionné (**0.1.16-draft** en fin d’epic), `operation_id` traçables (registre §3).
- **Artefacts utiles :** migrations Alembic **8.1–8.4**, pytest `test_story_8_*`, gates Story Runner documentés (registre §4), `_bmad-output/implementation-artifacts/tests/test-summary-story-8-*-e2e.md`.

### 1.2 Ce qui a bien fonctionné (systèmes et processus)

- **Réinvestissement du modèle Epic 6 / 7** : story finale (**8.7**) traitée comme **système de preuves** (AC explicites MockTransport vs distant, quatre colonnes), pas comme narrative de clôture trompeuse.
- **Honnêteté explicite sur Paheko réel** : l’epic s’est d’abord terminé en **AC6 option B** honnête, puis la suite post-epic a permis de fermer une **preuve distante minimale** : `PAHEKO_API_USER` / `PAHEKO_API_PASSWORD` deviennent le **canon** d’installation (Basic Auth, aligné doc Paheko), le slice Recyclique a été réaligné sur `POST /api/accounting/transaction`, et une clôture réelle a créé une écriture Paheko observable (`id=131`) avec état Recyclique `delivered` / `resolu`.
- **Progression incrémentale maîtrisée** : chaque story a étendu le slice sans casser le gate pytest agrégé documenté ; **8.4** / **8.5** / **8.6** ont enrichi contrat et observabilité admin sans renommer à la légère les `operationId`.
- **Alignement avec Epic 1** : rappels matrice **1.6**, contrat minimal sync, gaps endpoint Paheko nommés comme **dette** plutôt que comme “fait acquis”.

### 1.3 Difficultés et frottements (sans blâme)

- **Preuve “verte” ≠ preuve comptable** : risque permanent de confondre **65 tests pytest** (gate **8.7**) et **réconciliation observée dans Paheko** — le registre et **8.7** ont dû **verrouiller** ce vocabulaire.
- **Dépendance environnement réel** : la preuve réelle n’a été fermée qu’après injection d’identifiants hors dépôt et réalignement du slice sur l’endpoint officiel ; cela confirme que la CI seule ne suffisait pas.
- **Charge cognitive intégration** : idempotence, quarantaine, corrélation, politique A1 — surface API admin dense ; le **support** (timeline, filtres) devient indispensable pour l’exploitation, pas un luxe.
- **Nom de fichier registre** : suffixe `…-squelette` malgré document **consolidé** (même friction signalée en rétro Epic 7 pour le registre réception).

### 1.4 Apprentissages

1. **Pour toute intégration tierce “comptable”** : séparer dès le début **preuve Recyclique** (mocks, DB locale) et **preuve cible distante** ; sinon le vert CI devient une **fiction de réconciliation**.
2. **L’option (A) / (B) de l’AC6 (8.7)** est un **gabarit réutilisable** : soit procédure reproductible + capture, soit constat d’impossibilité + dette + owner backlog — **sans** équivaloir « story done » et « Paheko prouvé ».
3. **La continuité Epic 7 §2.3** (contrats sync, correspondances, sous-suites pytest) a **été utile** ; le point bloquant majeur était bien **terrain + secrets** hors repo jusqu’à la validation réelle.
4. **Les tests `MockTransport` sont une rampe**, pas une fin : ils valident le **comportement orchestrateur** Recyclique, pas la sémantique acceptée par l’API Paheko réelle.

- **Renommage ou alias documentaire** du fichier registre Epic 8 (comme proposé pour Epic 7) pour éviter l’ambiguïté `squelette` vs consolidé.

## Partie 2 — Préparation Epic 9 (et jalons voisins)

### 2.1 Dépendances satisfaites par Epic 8

- **Slice clôture** : outbox durable, retry, quarantaine, audit, corrélation, garde-fou **409** avant enqueue — base pour élargir d’autres `operation_type` **hors** epic sans refondre le noyau.
- **Discipline de preuve** : registre quatre colonnes + **AC6** — réutilisable pour tout nouveau connecteur ou flux sync.

### 2.2 Risques spécifiques Epic 9+

- **Élargissement scope sync** : risque d’**oublier** que la preuve réelle obtenue ne couvre que le **slice clôture** actuel — tout nouveau module doit **citer** l’état exact du registre Epic 8.
- **Modules déclaration / adhérents (Epic 9)** : ne pas **court-circuiter** l’outbox ni dupliquer les invariants **8.2–8.6** côté front.

### 2.3 Actions et accords proposés (SMART, sans estimation temporelle)

1. **Consolider AC6 option A** désormais obtenue — *critère :* tous les artefacts dérivés restent alignés sur le registre Epic 8 (preuve datée, version Paheko, écriture observée, limites restantes). **Owner suggéré :** dev backend + pilotage.

3. **Repasser `guide-pilotage-v2.md` §3** pour **Epic 8** (jalon, lien registre, mention explicite du passage **B → A** et des dettes résiduelles). **Owner suggéré :** pilotage.

4. **Décision documentée** sur `poste_reception.site_id` avant tout durcissement **mapping multi-site**. **Owner suggéré :** architecte + PO.

5. **Pour Epic 9 / create-story** : citer les `operation_id` Paheko-related existants et l’état réel du registre Epic 8 : **preuve minimale obtenue sur le slice clôture**, mais dettes résiduelles sur headers et modélisation comptable des écarts. **Owner suggéré :** SM + architecte.

### 2.4 Découverte significative pour la planification

Une **intégration comptable** peut être **techniquement livrée** (migrations, API, tests mockés, gates verts) tout en restant **métier non prouvée** côté tiers ; inversement, quand la preuve distante arrive après coup, la documentation doit être **reconsolidée partout** pour éviter de laisser cohabiter un récit B et un récit A.
