- La demo reste encore largement portee par une enveloppe de contexte mockee pour les parcours transverses.
- Les surfaces transverses livrees sont operables, mais les vraies donnees metier restent en attente des futurs `operation_id` OpenAPI et des slices data backend.

## 4. Continuite depuis Epic 4

La retro Epic 4 avait formalise quatre lecons :

1. expliciter le **lieu de preuve** des gates produit ;
2. separer puis reunir **preuve automatisee** et **preuve humaine** ;
3. traiter le **bundle servi** comme une responsabilite explicite ;
4. considerer le **fallback visible** comme un livrable produit.

**Application dans Epic 5 :**

- Ces lecons ont ete mieux appliquees globalement : les stories visibles ont fini par etre verifiees via DevTools sur `http://127.0.0.1:4444`.
- Le bundle servi est devenu un point de verification recurrent, pas un detail post-scriptum.
- Les fallbacks et etats degrades ont ete integres comme partie du shell transverse, pas comme dette cachee.
- Le point encore perfectible est la discipline d'orchestration autour des URLs de validation et de l'alignement immediat des statuts.

---

## 5. Lecons retenues

1. **Le shell transverse peut avancer loin sans recreer le metier** si la hierarchie de verite reste claire : OpenAPI -> `ContextEnvelope` -> `NavigationManifest` -> `PageManifest` -> prefs locales.
2. **La preuve servie doit etre un reflexe standard pour toute story UI visible**, pas une exception rattrapee en fin d'epic.
3. **Les templates/layouts partages reduisent vraiment le drift** entre dashboard, listings, consultation et admin.
4. **Documenter les gaps explicitement** vaut mieux que les masquer derriere des placeholders ambitieux.
5. **Le frontend legacy reste encore necessaire pour les flows riches**, et c'est acceptable tant que cette coexistence est nommee franchement.
