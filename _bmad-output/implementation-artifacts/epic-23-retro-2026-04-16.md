### 1.2 Ce qui a bien fonctionné

- **Découpage epic** : backend **23.1** puis Peintre **23.2 / 23.3** en parallèle possible une fois les contrats stables — réduit le couplage et clarifie les gates.
- **Tests ciblés** : peloton pytest documenté (`test-summary-story-23-1-qa-chain.md`) ; vitest par fichiers pour l’admin compta — feedback rapide vs batterie entière.
- **Revue code structurante** : **CR1 → correctifs → CR2** sur **23.2** et **23.3** a évité de laisser passer une incohérence **CREOS** (garde-fous page vs navigation).
- **Documentation contrat** : harmonisation des descriptions **OpenAPI** (références **22.3** + **23.2 / 23.3**) pour refléter à la fois la fondation API et les surfaces Peintre.
- **Honnêteté 22.8** : distinction maintenue entre preuve locale (mocks / pytest) et validation **Paheko réel** pour la colonne terrain.

### 1.3 Difficultés et frottements (système, pas personnes)

- **Orchestration BMAD** : risque de **Plan B** (tout dans un seul Task) quand les spawns imbriqués ne sont pas utilisés — la valeur du graphe CS→…→CR est la **traçabilité** ; à exiger quand l’équipe veut une preuve d’audit d’exécution.
- **Charge cognitive monorepo** : sans consigne explicite, les gates « pytest / npm test complets » font exploser le temps de feedback — la consigne « périmètre intelligent » doit être **répétée à chaque sous-agent**.
- **Symétrie CREOS entre surfaces sœurs** : **23.3** a montré qu’une nouvelle page manifestée peut diverger des **mêmes** `required_permission_keys` qu’une story voisine (**23.2**) — à traiter comme **checklist** dès qu’on ajoute une page admin sous le même hub.
- **Connexion / timeout** : interruption de Task parent sans rapport §7.1 — le **resume_at** + synthèse d’état dans le sprint/story limite la perte.

### 1.4 Apprentissages

1. **Checklist CREOS** pour toute nouvelle page admin sous un hub existant : aligner **page manifest**, **navigation-transverse-served**, **navigation.json** et **tests contract** sur les **mêmes** clés que les pages sœurs du même epic.
2. **Gates = périmètre nommé** dans le brief (fichiers de tests listés) ; défaut = dette de temps et fausse impression de « tout vert ».
3. **OpenAPI** : les descriptions sont un **contrat lisible humain** — les mettre à jour quand les stories livrent de nouvelles surfaces consommatrices (ici **23.2 / 23.3**).
4. **Rétro Epic 22** : si absente, prévoir une **rétro courte** ou un **artefact de continuité** après un epic « rail correctif » long, pour ne pas perdre les engagements process entre **22** et **23**.
