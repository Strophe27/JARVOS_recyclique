# Rétrospective Epic 17 + Vague Hotfix Terrain

**Date :** 2026-03-02  
**Scope :** Epic 17 (stories 17.0 à 17.11) + Vague Hotfix Terrain (HF-1 à HF-4 + ajustements manuels)  
**Statut :** Clôturé — tous les critères de done atteints

---

## 1. Résumé exécutif

Epic 17 a remédié en totalité aux 18 écarts identifiés lors de l'audit post-Epic 16, couvrant sécurité (RBAC, guards, contrôle d'accès), fonctionnalités admin précédemment en mode stub (BDD, import legacy, supervision, logs, rapports) et couverture de tests technique. Les 11 stories ont été exécutées en ordre strict de vagues (P0→P1→P2), avec un crash d'orchestrateur entre 17.6 et 17.7 traité sans perte. Une vague Hotfix Terrain de 4 stories a suivi immédiatement après les tests en conditions réelles, corrigeant 5 anomalies bloquantes/importantes (AuthGuard manquant, cookie de session absent, redirection post-login incorrecte, absence de bootstrap super_admin). Les 6 critères de smoke test terrain sont tous verts à clôture.

---

## 2. Ce qui a bien fonctionné

- **Ordre strict des vagues respecté.** V1 (P0) → V2 (P1) → V3 (P2) : aucune story P1 lancée avant que P0 soit solide. La discipline de sequencing a évité les effets de bord.
- **Densité de couverture de tests remarquable.** ~146 pytest + ~35 vitest ajoutés sur l'ensemble de l'epic, couvrant happy path et chemins d'erreur — dépassant l'objectif initial (story 17.11 seule = 41 pytest).
- **Fermeture traçable de tous les E16-\*.** Chaque écart de l'audit (E16-A-001 à E16-C-007, 18 IDs) est fermé avec preuve associée dans le journal. Aucun ID laissé ouvert ou clôturé sans artefact.
- **Récupération rapide sur crash orchestrateur.** Le crash entre 17.6 et 17.7 a été absorbé via reprise manuelle sans perte de données ni de contexte — le registre d'état a joué son rôle.
- **Réactivité post-tests terrain.** La vague HF a été planifiée, exécutée et clôturée le même jour (2026-03-02) que la découverte des anomalies AT-001 à AT-005. Zéro latence entre diagnostic et correction.
- **Ajustements manuels documentés.** Le sélecteur rôle `super_admin` dans les 3 pages admin, le patch `main.py` bootstrap et la promotion live `strophe_admin` ont été capturés dans le plan HF — aucune modification silencieuse.
- **Pipeline import legacy entièrement déstubifié.** Les 4 étapes (analyze / preview / validate / execute) sont opérationnelles avec assertions sur effets réels — confiance en migration legacy.
- **Script bootstrap `super_admin` idempotent.** L'exécuter deux fois ne produit pas d'erreur — bonne pratique pour les opérations de maintenance CLI.

---

## 3. Ce qui a moins bien fonctionné / points d'attention

- **Crash orchestrateur non anticipé.** Le crash entre 17.6 et 17.7 reste inexpliqué (charge ? timeout ? contexte trop long ?). La reprise a fonctionné mais a créé une interruption non prévue. Aucun mécanisme de checkpoint automatique n'existait à ce moment.
- **Anomalies terrain non détectées pendant Epic 17.** Les 5 anomalies AT (AuthGuard absent, cookie session, redirect post-login, bootstrap super_admin) auraient pu être détectées lors des tests de story 17.4 ou 17.1. La couverture tests automatiques n'a pas rattrapé l'absence de tests de parcours utilisateur complets (login → navigation → session persistante).
- **Stories 17.0 et 17.1 pré-existantes.** Ces deux stories étaient déjà done avant le début formel de l'epic. Leur présence dans le compte crée une légère surestimation de la vélocité observable.
- **Découverte tardive des routes non gardées.** Les routes `/dashboard`, `/caisse`, `/reception`, `/profil` sans AuthGuard existaient depuis plusieurs epics (hors scope auditié par E16). L'audit E16 ciblait le backend et les stubs — le garde-fou frontend de base a failli passer entre les mailles.
- **Ajustements manuels hors story.** Le patch `main.py` bootstrap et la promotion live `strophe_admin` ont été réalisés manuellement en dehors du cycle story. Leur capture dans le plan HF est bonne, mais leur traçabilité dans le code (commit nominatif) reste à vérifier.
- **Vague HF non planifiée dans l'epic initial.** Le budget de l'epic ne prévoyait pas de vague hotfix post-terrain — normal pour un premier exercice de ce type, mais à anticiper dans les prochains epics (réserver ~10-15% de capacité pour correctifs post-smoke).

---

## 4. Leçons apprises (actionnables)

1. **Intégrer un smoke test terrain systématique en fin d'epic.** Formaliser une checklist de parcours utilisateur réels (connexion → navigation → F5 → déconnexion) avant de déclarer un epic done. La vague HF montre que les tests automatiques unitaires ne remplacent pas un parcours manuel de 10 minutes.

2. **Ajouter un test AuthGuard dans chaque story qui touche au routeur.** Toute modification d'`App.tsx` doit inclure un test de redirection non-auth sur les routes concernées. Règle à documenter dans la checklist create-story.

3. **Réserver une session de contexte dédiée aux stories longues.** Le crash orchestrateur est possiblement lié à un contexte trop chargé. Pour les stories > 3 fichiers modifiés, démarrer une session fraîche plutôt que d'enchaîner dans le même thread.

4. **Documenter les ajustements manuels comme micro-stories.** Même un patch d'une ligne (`main.py`) mérite un commit conventionnel traçable. Créer un template "micro-fix" pour les corrections < 30 min qui ne méritent pas une story complète mais doivent rester visibles dans le git log.

5. **Vérifier la cohérence cookie/session à chaque story d'auth.** La story 17.2 (contrôle d'accès `/v1/users/me`) et la story 17.1 (RBAC) étaient les bons endroits pour détecter que le login legacy ne posait pas de cookie. Ajouter à la checklist : "Est-ce que cette story touche au flux auth ? → vérifier que login legacy et SSO posent tous deux le cookie de session."

6. **Capitaliser sur le registre anomalies terrain.** Le fichier `17-z-registre-anomalies-terrain.md` a été très efficace : capture rapide → diagnostic → plan HF le jour même. Utiliser ce patron pour tout epic qui touche à des flows utilisateur en production.
