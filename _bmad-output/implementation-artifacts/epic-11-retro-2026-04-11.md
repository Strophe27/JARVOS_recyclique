**Ligne directrice tenue (système) :** autorité backend, manifests reviewables, pas de re-spécification métier caisse (Epic 6 reste la référence métier).

### 1.2 Ce qui a bien fonctionné

- **Méthode legacy-first formalisée** : artefacts `_02_`, `_03_`, `_04_` du 2026-04-10 comme fil conducteur create-story / dev-story.
- **Décisions d'alignement routier écrites** (ex. **11.2** : canon `/dashboard` vs legacy `/`).
- **Gouvernance CREOS** : refus de pages one-off finales hors manifestes ; extensions via navigation / page manifests.
- **Automatisation Peintre** : tests Vitest ciblés sur les stories, gates verts documentés dans les story files.
- **Séparation des rôles epic** : Epic 11 UI observable vs Epic 6 domaine caisse — pas de dérive métier dans le front.

### 1.3 Difficultés et frottements (focus système, pas personnes)

- **Ambiguïté « observable » vs « parité terrain »** : le vocabulaire de la story **11.3** et des AC peut laisser croire qu'une **preuve technique** suffit pour une **preuve humaine** de parité — ce n'est pas le cas ici.
- **Slice CREOS `/caisse` vs route legacy kiosque** : l'écart est documenté, mais le **ressenti legacy** n'est pas reproduit au niveau attendu pour une validation terrain.
- **Dépendance au pilote démo / runtime** : une partie du chemin **11.3** vit dans des mécanismes de démo (`RuntimeDemoApp`, alias) — utile pour l'ingénierie, **fragile** comme message « c'est prêt pour le terrain » sans couche de validation utilisateur.
- **Pas de rétro Epic 10** : peu de fil conducteur formel « industrialisation → pilote UI » dans les archives BMAD, même si le YAML montre un cheminement parallèle.

### 1.4 Apprentissages

1. **Distinguer trois niveaux de preuve** : (a) build / tests automatisés, (b) preuves DevTools / matrice, (c) **validation humaine terrain** (parité perçue). Ne pas conclure (c) à partir de (a) seul ou (a)+(b) insuffisant.
2. **Pour toute story « parité UI legacy »** : si le livrable repose sur un **alias** ou un **bundle** existant sans recomposition visuelle, l'**intitulé de story** et les **AC** doivent dire explicitement « socle routage + chrome » vs « parité opérateur ».
3. **La clôture `done` au YAML** doit rester **alignée avec une phrase honnête** dans le story file sur ce qui est **réellement** validé — évite la dette narrative pour les epics **12–14**.
4. **Les Epics 12–14** sont le bon **contenant** pour reprendre la parité étendue : le constat sur **11.3** devient une **entrée de planification**, pas un échec personnel.
