### 1.2 Ce qui a bien fonctionné

- **Cadrage par l’audit** : tableau §9 **P0–P2** mappé aux stories — peu de dérive de périmètre si on lit `epics.md` § Epic 26.
- **Normes dans le dépôt** : conventions **ORM sync** et **PEP 604** versionnées ; **ADR** pour le guide tests évite l’attente d’un artefact fantôme.
- **Gates reproductibles** : `compileall`, `test_infrastructure.py`, pelotons nommés dans les stories et dans **`test-summary.md`**.
- **Clôture transverse** : `epic-26-cloture-f7-f11-trace.md` matérialise **F7–F11** hors lignes P* — distinction **traité / reporté** explicite.
- **Coordination P2** : parallélisme **26.5** vs **26.3/26.4** autorisé quand périmètres disjoints — réduit les conflits Git inutiles.

### 1.3 Difficultés et frottements (système, pas personnes)

- **Charge cognitive « conventions »** : trois couches en parallèle (ORM sync, PEP 604, ruff) — les revues doivent **citer** la bonne section (`architecture/index.md` § Epic 26).
- **Boucle CR sur 26.2** : signal que **doc + liste de tests + intention AC** doivent être alignés **avant** premier passage CR quand le sujet est sensible (nom **AdminService** interdit).
- **Boucle QA sur 26.4** : migrations typage **Pydantic** peuvent exiger un **second regard** sur les assertions — prévoir que le premier run QA ne soit pas toujours le dernier sur les schémas.
- **Findings reportés** : **F7** isolation tests, **F8** docstrings à grande échelle — **acceptés comme reportés** avec propriétaire / convergence ; risque que de nouvelles PR **élargissent** le dette si la DoD story n’est pas relue fichier par fichier.

### 1.4 Apprentissages

1. **Un epic « qualité API » peut rester fin** si chaque story référence un **finding** ou une **ligne P*** du tableau audit — évite le refactor « par feeling ».
2. **La première vague PEP 604** (`26.4`) établit la règle : toute PR future qui touche un schéma doit **finaliser** `T | None` dans le fichier édité — continuité sans migration massive unique.
3. **Ruff + Black/isort** documentés dans `pyproject` et README — l’outillage **P2** n’est pas optionnel tribalement ; il est **vérifiable**.
4. **Après suppression `AdminService`**, toute couche admin async future = **nouveau module + revue** — la règle dans `epics.md` évite le revert silencieux.
